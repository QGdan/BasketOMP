# 已知问题

## KI-001：本机 MinGW 不含 Sanitizer 运行库

- 现象：使用 `-fsanitize=address,undefined` 链接时找不到 `libasan` 和 `libubsan`。
- 影响：本机不能使用 ASan/UBSan 自动检测内存问题。
- 当前措施：启用完整编译警告、Debug 断言、边界检查、重复运行、结构不变量检查和串并行逐项比较。
- 后续方案：若在带 Sanitizer 的 Linux/GCC 环境复现项目，再补跑完整测试。

## KI-002：未自动记录 large 峰值工作集

- 现象：large 串行和 8 线程均完成，但当前基准脚本只记录阶段耗时，没有轮询进程 `PeakWorkingSet64`。
- 影响：可以证明在 27.81 GiB 机器上可运行，但报告中不能给出精确峰值内存数字。
- 当前措施：记录数据规模、唯一边数、机器总内存和完整运行成功事实，不虚构峰值。
- 后续方案：若报告必须展示峰值内存，可在基准脚本中加入独立进程监控后补跑一次 large 8 线程。

## KI-003：large 串行不适合作为日常回归

- 现象：完整串行算法约 1,218 秒，其中推荐约占 98.2%。
- 影响：开发中频繁运行 large 成本过高。
- 当前措施：toy/small 用于正确性，medium 用于重复性能实验；large 仅作最终规模验证。

## KI-004：精确归一化分母增加完整图邻接项大小

- 现象：为保持优化前推荐 ID 校验和，`CooccurNeighbor` 保存 `double normalization_denominator`，结构由 8 字节增加到 16 字节。
- 影响：完整 large CSR 图的邻接数组理论大小约从 0.61 GiB 增至 1.21 GiB；构图阶段仍会先创建完整图。
- 当前措施：最终推荐配置使用 Top-50，压缩后 large 仅保留 2,449,985 个邻接项；完整图模式仍用于兼容和基线实验。
- 后续方案：若必须进一步降低瞬时峰值，可研究分商品流式 Top-N，但该实现复杂度不属于当前版本。

## KI-005：Top-N 后相同算法的并行效率下降

- 原现象：O6 实验中，medium Top-50 OpenMP 8 线程相对同配置串行加速比仅为 1.310。
- 原因：推荐阶段已从数秒降到几十毫秒，串行构图与局部表归并占比显著上升，符合 Amdahl 定律。
- 当前措施：已实现两遍式并行邻接构建、按商品并行 Top-N 和无锁分桶并行归并；2026-07-18 medium 本地诊断中，8线程64桶将归并中位数从189 ms降至27 ms，并把总算法时间从397 ms降至242 ms。
- 后续方案：学校平台仍需在 large 24/32/48线程比较128/256/384桶，不能把本地结果或先前收益估算直接写成最终结论。
## KI-006：受限环境下 manifest 的部分硬件字段可能为空

- 现象：部分 Windows 环境拒绝 `Get-CimInstance`，无法自动读取物理核心数和总内存。
- 当前措施：`collect_environment.ps1` 自动降级为注册表 CPU 名称、.NET 操作系统版本和逻辑线程数，不让实验失败。
- 后续方案：报告最终定稿时可由实验人员人工核对 manifest 中的空字段，但不得修改 raw CSV 性能记录。
