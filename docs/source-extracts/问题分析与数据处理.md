5. 问题建模分析与扩展

5.1 原始问题建模回顾

本项目将 Kaggle Instacart 竞赛的原始问题——"预测用户下一次购物篮中会包含哪些商品"——转化为一个基于商品共现统计的 Top-K 推荐问题。其核心建模逻辑如下：

用户建模：将每个用户视为一个按时间排列的历史购买序列；

购物篮建模：将每个订单视为一个包含多种商品的购物篮；

共现关系建模：在同一购物篮中共同出现的任意两种商品之间建立共现关系；

关联强度建模：以商品对在全体订单中的共现频次作为关联强度度量；

推荐生成建模：对用户历史购买商品及最近购物篮中的商品，查询共现图计算候选商品得分，输出 Top-K 推荐列表。

5.2 问题建模分析

5.2.1 建模思想分析

本项目采用的共现图模型本质上是一个无向加权图。图中节点代表商品，边权重代表两种商品在同一订单中共同出现的次数。推荐时，将用户历史购买的商品作为"查询起点"，在共现图中查找与其关联紧密的其他商品，从而得到候选推荐列表。

这种建模方式的核心思想是"物以类聚"——如果大量用户在购买香蕉的同时也购买了牛奶，那么这两种商品之间就存在较强的关联，当某个用户买了香蕉时，系统就可以向他推荐牛奶。与复杂的机器学习模型相比，共现统计方法原理直观、实现简单，且计算过程高度规则化，非常适合作为并行程序设计的实验对象。

5.2.2 建模合理性

（1）算法简单，易于验证正确性

共现统计的逻辑非常直观：遍历所有订单，对每个订单内的商品做两两组合，累加计数即可。串行版本代码量少、逻辑清晰，便于作为并行版本的正确性基准。调试时可以通过小规模数据人工验算，确保并行结果与串行结果完全一致。

（2）计算密集，适合展示并行优化效果

共现统计的时间复杂度与订单数量和每单商品数量密切相关。Instacart 数据集包含超过 300 万订单，部分订单包含数十种商品，两两组合后计算量巨大，串行执行耗时明显。这为 OpenMP 并行优化提供了充足的加速空间，符合课程"并行程序设计"的核心诉求。

（3）数据独立性好，适合 OpenMP 并行

不同订单之间的共现统计是相互独立的——计算订单 A 的共现关系不需要知道订单 B 的结果。因此可以将订单集合划分为若干份，分配给不同线程分别处理，每个线程维护自己的局部统计结果，最后统一合并。这种"分而治之"的策略完美契合 OpenMP 的并行模式。

（4）便于分析并行中的关键问题

在并行实现过程中，可以清晰地展示和讨论课程关注的几个核心问题：负载均衡（不同订单大小差异导致计算量不均）、线程局部存储与归并开销、临界区竞争（如果直接写全局表）等。这些分析点能够体现对并行程序设计的深入理解。

5.2.3 建模局限性

尽管上述建模在课程场景下具有合理性，但仍存在以下不足：

（1）未考虑时间衰减

模型将用户所有历史订单视为同等重要。但实际生活中，用户上周购买的商品对本次推荐的参考价值，显然高于半年前的购买记录。当前建模缺乏对时间近因的加权处理。

（2）冷启动问题

对于新用户（历史购买极少）或新商品（共现记录极少），共现图无法提供有效推荐，此时模型只能退化为按全局流行度排序，推荐效果会明显下降。

（3）共现矩阵稀疏

数据集中约有 5 万种商品，理论上商品对数量非常庞大，但实际有共现记录的商品对只占极小比例。如果存储方式不当，会造成大量内存浪费；如果采用稀疏存储，则需要设计高效的数据结构。

（5）未利用商品属性信息

数据集中提供了商品的货架类别（aisle）和部门类别（department）信息，但当前建模完全基于商品 ID 的共现统计，没有利用这些属性信息。例如，同一类别的商品往往具有替代关系，不同类别但经常搭配的商品具有互补关系，这些信息可以辅助提升推荐效果。

5.3 可考虑的扩展（可适当考虑添加）

针对上述局限性，结合"并行程序设计"课程以性能分析为核心的特点，建议做以下扩展，既能提升推荐效果，又能丰富并行设计的讨论内容：

扩展 1：引入时间衰减因子

思路：对历史订单按时间远近赋予不同权重，最近订单中的商品共现获得更高权重。

实现方式：在 orders.csv 中，order_number 字段表示该订单是用户的第几次购买，可近似作为时间顺序。共现统计时，权重不再简单加 1，而是累加 ，其中  为时间间隔。

对并行的影响：计算量略有增加，但不影响整体并行结构，仍可按订单并行。可以在报告中对比"等权共现"与"时权共现"的并行效率差异。

扩展 2：融合商品类别信息

思路：在共现图之外，利用商品的类别信息。若商品  与用户历史商品  属于同一类别或经常出现在同一类别的购买记录中，给予额外加分。

实现方式：利用 products.csv 中的 aisle_id 和 department_id，在评分函数中增加类别亲和度项。

对并行的影响：类别亲和度计算可与共现计算并行执行，为报告增加"多阶段并行"的分析内容。

扩展 3：Top-N 邻居截断

思路：不为每个商品保存所有共现邻居，仅保留共现次数最高的前  个邻居。

实现方式：共现统计完成后，对每个商品按共现权重排序，截断尾部；推荐阶段只需遍历 Top-N 邻居。

对并行的影响：截断操作需要排序，可讨论并行排序算法；同时内存占用显著下降，可对比不同  值下的内存与效率权衡。

扩展 4：引入负采样

思路：原始数据中只有"用户买了什么"的正样本。在验证推荐效果时，可随机采样用户未购买的商品作为负例，使评估指标更具区分度。

实现方式：对每个用户，从其未购买过的商品中随机选取若干作为负样本，检验推荐模型能否将正样本排在负样本前面。

对并行的影响：负采样过程可按用户并行，与主推荐计算形成第二个并行点，丰富"多粒度并行"的分析。

5.5 小结

本项目的问题建模以商品共现图为核心，抓住了购物篮推荐问题的本质关联规律，同时天然适合 OpenMP 并行化。其局限性主要体现在时间动态性、冷启动和稀疏性方面。通过引入时间衰减、类别信息和Top-N 截断等扩展，可以在不显著增加实现复杂度的前提下，提升推荐效果与并行设计的讨论深度，使课程报告在"算法合理性"与"并行性能分析"两个维度上都更加充实。

路径：~/basket_project/data/{small,medium,large}/

每个目录包含 4 个 CSV：

1. orders.csv

字段：order_id,user_id,eval_set,order_number,order_dow,order_hour_of_day,days_since_prior_order

字段：order_id,product_id,add_to_cart_order,reordered

说明：历史订单商品明细，用于统计共现关系

字段：同 prior

说明：验证集，用于计算 Hit Rate@K 和 Recall@K

4. products.csv

字段：product_id,product_name,aisle_id,department_id

说明：商品信息，含 aisle/department 可做类别扩展

数据清洗干了什么

原始数据一般比较干净，但你需要做以下检查和处理：

去重：检查 orders 和 order_products 是否有重复行，有则删除。

缺失值处理：days_since_prior_order 可能有 NaN（用户的首单），可填 0 或保留。

类型优化：把 order_id, product_id, user_id 等转为整型，节省内存。

一致性检查：确保 order_products__prior 里的所有 order_id 都能在 orders 中找到。

脚本代码

import pandas as pd

import os

os.makedirs('data/cleaned', exist_ok=True)

# 1. 读取

orders = pd.read_csv('data/raw/orders.csv')

order_prior = pd.read_csv('data/raw/order_products__prior.csv')

order_train = pd.read_csv('data/raw/order_products__train.csv')

products = pd.read_csv('data/raw/products.csv')

# 2. 去重

orders = orders.drop_duplicates()

order_prior = order_prior.drop_duplicates()

order_train = order_train.drop_duplicates()

# 3. 缺失值（首单的 days_since_prior_order 填0）

orders['days_since_prior_order'] = orders['days_since_prior_order'].fillna(0)

# 4. 类型优化

for col in ['order_id', 'user_id', 'product_id']:

if col in orders.columns:

orders[col] = orders[col].astype('int32')

if col in order_prior.columns:

order_prior[col] = order_prior[col].astype('int32')

if col in order_train.columns:

order_train[col] = order_train[col].astype('int32')

# 5. 一致性检查

prior_ids = set(order_prior['order_id'])

train_ids = set(order_train['order_id'])

order_ids = set(orders['order_id'])

print(f"prior中缺失的order_id: {len(prior_ids - order_ids)}")

print(f"train中缺失的order_id: {len(train_ids - order_ids)}")

# 6. 保存清洗后数据

orders.to_csv('data/cleaned/orders.csv', index=False)

order_prior.to_csv('data/cleaned/order_products__prior.csv', index=False)

order_train.to_csv('data/cleaned/order_products__train.csv', index=False)

products.to_csv('data/cleaned/products.csv', index=False)

print("清洗完成，保存在 data/cleaned/")
